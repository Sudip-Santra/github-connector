# GitHub Cloud Connector

A REST API cloud connector that integrates with the GitHub API using **OAuth 2.0** authentication. Built with **Python** and **FastAPI**, this application allows users to authenticate via GitHub and perform repository operations — fetch repositories, manage issues, view commits, and create pull requests — all through a clean, secure, and well-structured API.

## About the Project

This project is a **GitHub Cloud Connector** — a backend service that acts as a bridge between your application and the GitHub API. Instead of directly interacting with GitHub's API, users authenticate through this connector via OAuth 2.0 and then use simple REST endpoints to perform actions on their GitHub account.

**What it does:**
- Authenticates users securely via GitHub OAuth 2.0 (no passwords or tokens are stored in code)
- Provides REST API endpoints to interact with the user's GitHub data
- Handles pagination, error cases, and input validation robustly
- All operations are performed on behalf of the authenticated user using their OAuth token

**Why OAuth 2.0 over Personal Access Tokens (PAT):**
- Users don't need to manually generate and share tokens
- Tokens are scoped — the app only gets the permissions it asks for
- Tokens can be revoked by the user at any time from their GitHub settings
- More secure — secrets are exchanged server-side, never exposed to the client

|---|---|

| `config.py` | Single source of truth for all environment variables. Every module imports from here instead of reading `.env` directly. |

| `api/auth.py` | Handles the complete OAuth 2.0 flow: redirect to GitHub, exchange code for token, create JWT, set cookie. |

## GitHub OAuth App Setup (Step-by-Step)

You need to create a GitHub OAuth App to get the `Client ID` and `Client Secret`. Follow these steps carefully:

### Step 1: Go to GitHub Developer Settings

1. Log in to your GitHub account.
2. Click on your **profile picture** (top-right corner) → **Settings**.
3. Scroll down in the left sidebar and click **Developer settings** (at the very bottom).
4. Click **OAuth Apps** in the left sidebar.
5. Click the **"New OAuth App"** button.

### Step 2: Fill in the Application Details

| Field | Value | Explanation |
|---|---|---|
| **Application name** | `Connector GitHub` | Any name you want — this is shown to users during authorization (GitHub doesn't allow app name starting with GitHub or Gist) |
| **Homepage URL** | `http://localhost:8000` | The URL where your app runs locally |
| **Application description** | `A cloud connector to interact with GitHub API` | Optional — describes what your app does |
| **Authorization callback URL** | `http://localhost:8000/callback` | **Critical** — this must exactly match `GITHUB_REDIRECT_URI` in your `.env` file. This is where GitHub sends the user back after authorization. |

### Step 3: Register the Application

1. Click **"Register application"**.
2. You will be taken to your app's settings page.
3. You will see your **Client ID** displayed on the page — **copy it**.

### Step 4: Generate a Client Secret

1. On the same app settings page, click **"Generate a new client secret"**.
2. **Copy the secret immediately** — GitHub will only show it once. If you lose it, you'll need to generate a new one.

### Step 5: Generate a JWT Secret Key

You need a random secret key for signing JWT tokens. Run this command in your terminal:

**On Windows (PowerShell):**
```powershell
python -c "import secrets; print(secrets.token_hex(32))"
```

**On macOS/Linux:**
```bash
python3 -c "import secrets; print(secrets.token_hex(32))"
```

**Using OpenSSL (any platform):**
```bash
openssl rand -hex 32
```

Any of the above commands will output a 64-character random hex string like:
```
8c6d5261fcb08570d907b6150d61e6e6bd3a313500bad9fddd28f841641fa684
```

Copy this value — you'll use it as `JWT_SECRET` in the next step.

### Step 6: Summary of Values

After completing the above steps, you should have:

| Value | Where to Find It | Example |
|---|---|---|
| `GITHUB_CLIENT_ID` | GitHub OAuth App settings page | `Ov23lixxxxxxxxxx` |
| `GITHUB_CLIENT_SECRET` | Generated in Step 4 (shown once) | `2fb95f74xxxxxxxxxx` |
| `JWT_SECRET` | Generated in Step 5 | `8c6d5261fcb08570...` |

## Local Setup Guide

### Step 1: Clone the Repository

```bash
git clone https://github.com/Sudip-Santra/github-connector.git
cd github-connector
```

### Step 2: Create a Virtual Environment

**On Windows:**
```bash
python -m venv venv
venv\Scripts\activate
```

**On macOS/Linux:**
```bash
python3 -m venv venv
source venv/bin/activate
```

### Step 3: Install Dependencies

```bash
pip install -r requirements.txt
```

This installs:
| Package | Purpose |
|---|---|
| `fastapi` | Web framework for building the REST API |
| `uvicorn` | ASGI server to run the FastAPI application |
| `python-dotenv` | Loads environment variables from `.env` file |
| `httpx` | Async HTTP client for making requests to GitHub API |
| `pyjwt` | Creates and verifies JWT tokens for session management |

### Step 4: Configure Environment Variables

1. Copy the example environment file:

   **On Windows:**
   ```bash
   copy .env.example .env
   ```

   **On macOS/Linux:**
   ```bash
   cp .env.example .env
   ```

2. Open `.env` in your editor and fill in your values:

   ```env
   # HOST and PORT for the FastAPI application
   HOST=localhost
   PORT=8000

   # GitHub OAuth Credentials (from GitHub Developer Settings)
   GITHUB_CLIENT_ID=<paste your Client ID here>
   GITHUB_CLIENT_SECRET=<paste your Client Secret here>
   GITHUB_REDIRECT_URI=http://localhost:8000/callback

   # GitHub API
   GITHUB_API_BASE_URL=https://api.github.com

   # Security Keys
   JWT_SECRET=<paste your generated JWT secret here>
   ALGORITHM=HS256
   ```

   | Variable | What to Put |
   |---|---|
   | `HOST` | `localhost` (keep as-is for local development) |
   | `PORT` | `8000` (keep as-is, or change if port is in use) |
   | `GITHUB_CLIENT_ID` | Your OAuth App's Client ID from [Step 3](#step-3-register-the-application) |
   | `GITHUB_CLIENT_SECRET` | Your OAuth App's Client Secret from [Step 4](#step-4-generate-a-client-secret) |
   | `GITHUB_REDIRECT_URI` | `http://localhost:8000/callback` (must match what you registered on GitHub) |
   | `GITHUB_API_BASE_URL` | `https://api.github.com` (GitHub's API base URL) |
   | `JWT_SECRET` | Your random secret key from [Step 5](#step-5-generate-a-jwt-secret-key) |
   | `ALGORITHM` | `HS256` (JWT signing algorithm) |

### Access the Application

| URL | Description |
|---|---|
| `http://localhost:8000/docs` | **Swagger UI** — Interactive API documentation where you can test all endpoints |
| `http://localhost:8000/redoc` | **ReDoc** — Alternative API documentation (read-only) |
| `http://localhost:8000/health` | Health check endpoint |

### Test the Full Flow

1. Open `http://localhost:8000/docs` in your browser.
2. Click on the `GET /login` endpoint → Click **"Try it out"** → Click **"Execute"**.
3. You'll be redirected to GitHub's authorization page — click **"Authorize"**.
4. GitHub redirects back to `/callback` — you'll see a success response with your username.
5. The `auth_token` cookie is now set in your browser.
6. Now you can test any protected endpoint (e.g., `GET /repos`, `GET /list-issues`, etc.) directly from Swagger UI — the cookie is sent automatically.

## How Authentication Works

This application uses the **OAuth 2.0 Authorization Code flow**:

```
User                    Our Server                 GitHub
  │                         │                         │
  │  1. GET /login          │                         │
  │────────────────────────>│                         │
  │                         │                         │
  │  2. Redirect to GitHub  │                         │
  │<────────────────────────│                         │
  │                         │                         │
  │  3. Authorize on GitHub                           │
  │──────────────────────────────────────────────────>│
  │                         │                         │
  │  4. Redirect to /callback?code=xxx                │
  │<──────────────────────────────────────────────────│
  │                         │                         │
  │  5. GET /callback       │                         │
  │────────────────────────>│                         │
  │                         │  6. Exchange code       │
  │                         │  for access token       │
  │                         │────────────────────────>│
  │                         │                         │
  │                         │  7. Access token        │
  │                         │<────────────────────────│
  │                         │                         │
  │                         │  8. Fetch user profile  │
  │                         │────────────────────────>│
  │                         │                         │
  │                         │  9. User data           │
  │                         │<────────────────────────│
  │                         │                         │
  │  10. Set auth_token     │                         │
  │  cookie (JWT, 7 days)   │                         │
  │<────────────────────────│                         │
  │                         │                         │
  │  11. Subsequent API     │                         │
  │  calls with cookie      │                         │
  │────────────────────────>│  12. GitHub API calls   │
  │                         │  using stored token     │
  │                         │────────────────────────>│
```

**Key points:**
- The GitHub access token is stored **inside the JWT** (not in a database).
- The JWT is set as an **HTTP-only cookie** so JavaScript cannot access it.
- The cookie expires after **7 days**, after which the user must re-login.
- Every protected endpoint extracts the GitHub token from the JWT to make API calls on behalf of the user.

For a more detailed breakdown, see [docs/auth-implementation.md](docs/auth-implementation.md).

## Security Measures

| Measure | Implementation |
|---|---|
| **No hardcoded secrets** | All credentials are loaded from `.env` via `config.py`. The `.env` file is in `.gitignore`. |
| **HTTP-only cookies** | The `auth_token` cookie has `httponly=True`, preventing JavaScript access (XSS protection). |
| **SameSite cookie** | The cookie uses `samesite=lax`, providing CSRF protection. |
| **JWT signed tokens** | Tokens are signed with `HS256` using a secret key — tampered tokens are rejected. |
| **Token expiration** | JWT tokens expire after 7 days. Expired tokens are automatically rejected. |
| **Server-side token exchange** | The `client_secret` is never sent to the client — code-to-token exchange happens server-side. |
| **Input validation** | All request bodies are validated using Pydantic models with type checking and constraints. |
| **Scoped permissions** | The OAuth app only requests the `repo` scope — no unnecessary permissions. |
| **`.env.example` provided** | A template file is included so developers know which variables to set without exposing real values. |

## Error Handling

All endpoints return consistent error responses:

```json
{
    "detail": "Human-readable error message"
}
```

| Status Code | Meaning | Example |
|---|---|---|
| `400` | Bad Request | OAuth code exchange failed |
| `401` | Unauthorized | Missing/expired/invalid auth cookie |
| `403` | Forbidden | No permission for this action |
| `404` | Not Found | Repository does not exist |
| `409` | Conflict | Repository is empty (no commits) |
| `410` | Gone | Issues are disabled for the repository |
| `422` | Validation Error | Invalid input, duplicate PR, branch not found |
